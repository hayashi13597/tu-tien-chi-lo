# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A cultivation-game (gameplay rebuilt to 100% feature parity with Nhất Niệm Tiêu Dao / 一念逍遥) with a Node/Express backend (`backend/`) and a Next.js frontend (`frontend/`). See `docs/superpowers/specs/` for design docs and `docs/superpowers/plans/` for implementation plans — read the relevant spec before changing game-logic behavior.

## Mandatory Rules

- **Architecture: Clean Architecture.** Backend code is organized into strict layers with dependencies pointing inward only:
  - `domain/` — entities and pure business logic (e.g. linh khí accumulation formula, breakthrough pity/success-rate formula, stage transitions). Zero framework or library dependencies. Defines repository/service interfaces (ports) that outer layers implement.
  - `application/` — use cases that orchestrate domain logic (e.g. RegisterUser, LoginUser, GetCultivationState, AttemptBreakthrough). Depends only on domain interfaces, never on concrete infrastructure.
  - `infrastructure/` — concrete implementations of domain ports: Prisma-backed repositories, JWT token service, bcrypt password hasher, realm config data.
  - `presentation/` — Express routes, controllers, middleware, request validation/DTO mapping. Translates HTTP ↔ use case input/output.
  - A composition root (`src/main.ts` or `app.ts`) wires concrete infrastructure into use cases and controllers.
  - `domain` must never import from `infrastructure` or `presentation`.
- **Comment logic clearly.** Non-trivial business/domain logic (formulas, state transitions, concurrency handling, etc.) must have clear comments explaining the *why* and the mechanics — do not leave complex logic uncommented.
- **Update CLAUDE.md after every task.** When executing an implementation plan, after completing each task, update this file to reflect new architecture, commands, or notes introduced by that task.
- **Use context7 (`ctx7` CLI) before writing library-specific code.** Before using any API from a dependency (Express, Prisma, Next.js, GSAP, jsonwebtoken, zod, etc.), fetch current docs for that library via context7 and cross-check against the exact version pinned in `package.json` — do not rely on training data for library API shape.

## Phase 2: Backend Auth Cookie Upgrade

Task 1: extended `TokenService` port with `signRefreshToken`/`verifyRefreshToken`; `JwtTokenService` now takes two secrets (`accessSecret`, `refreshSecret`) and signs access tokens with a 15-minute expiry (down from Phase 1's 7 days) and refresh tokens with a 7-day expiry. `FakeTokenService`'s token-string prefixes changed to `access-token-for-`/`refresh-token-for-` to distinguish token kinds the same way the real service's two secrets do. This breaks `tests/unit/LoginUserUseCase.test.ts`'s hardcoded literal (fixed in Task 3) — `tests/unit/auth.middleware.test.ts` turned out to already be prefix-agnostic (it round-trips via `signAccessToken`, never hardcodes the literal), so it was unaffected by this change despite the original plan expecting it to break too; Task 5 still updates that file for the cookie-fallback feature itself.

Task 2: added `cookie-parser`/`cors` dependencies (`package.json`) and `presentation/cookies.ts` (`setAuthCookies`/`clearAuthCookies`), centralizing cookie attributes (httpOnly, sameSite=lax, path=/) so all 4 auth call sites (register/login/refresh/logout) stay in sync. Access token max-age is 15 minutes, refresh token max-age is 7 days; `clearAuthCookies` explicitly passes `path: '/'` to Express's `clearCookie` so the browser recognizes it as the same cookie and actually deletes it (not auto-matched). Unit-tested with 3 tests (httpOnly/path/sameSite attributes, max-age values, clear behavior).

Task 3: `RegisterUserUseCase` now takes `(users: UserRepository, passwordHasher: PasswordHasher, tokenService: TokenService)` and returns `RegisterUserOutput = { id, username, accessToken, refreshToken }` — register also logs the user in. `LoginUserUseCase.execute()` now returns `LoginUserOutput = { token, refreshToken }`. Both use cases' tests updated with new token assertions (access/refresh prefix checks). `app.ts` updated to pass `tokenService` to `RegisterUserUseCase` and to `JwtTokenService`'s two-secret constructor as `new JwtTokenService(process.env.JWT_SECRET as string, process.env.JWT_REFRESH_SECRET as string)` — `JWT_SECRET` deliberately keeps its Phase 1 name unchanged (an existing deployment's env var keeps working), only `JWT_REFRESH_SECRET` is newly added. (Task review caught an initial implementation that renamed `JWT_SECRET` to `JWT_ACCESS_SECRET`; reverted to preserve that continuity guarantee from the approved design spec.) All unit tests pass; typecheck clean.

Task 4: added `RefreshAccessTokenUseCase` (`src/application/`, sliding renewal — issues a brand new refresh token on every successful refresh, not a re-signed copy) and mapped `INVALID_REFRESH_TOKEN` to 401 in `errorHandler`'s status map. `RefreshAccessTokenUseCase` verifies the input refresh token via `tokenService.verifyRefreshToken()`, catching any exception and throwing `DomainError('INVALID_REFRESH_TOKEN', ...)` to centralize error handling. On success, issues genuinely new access and refresh tokens (sliding window). Tested with 3 unit cases (valid refresh, invalid token, access token as refresh). `errorHandler.test.ts` extended with 4 tests total, new case confirms 401 status for `INVALID_REFRESH_TOKEN` code.

Task 5: `requireAuth` middleware now checks `req.cookies?.access_token` first, falling back to the `Authorization: Bearer` header if the cookie is absent — additive change, existing header-only callers (Phase 1 clients) still work unmodified. When both cookie and header are present, the cookie takes precedence (intended: browser clients always send the cookie once logged in, so a stale manual header cannot shadow the real session). `cookie-parser` is already a dependency (added in Task 2) but is NOT yet wired into the production app (that's Task 6's job); tests mount `cookie-parser` in their own test app. Test suite expanded from 3 to 7 cases: (1) reject no token, (2) reject invalid header, (3) reject invalid cookie, (4) allow valid header (backwards compat), (5) allow valid cookie, (6) prefer cookie over header, (7) reject refresh token as access token. All unit tests pass.

Task 6 (Phase 2): wired cookie-based auth end-to-end. `POST /auth/register` and `POST /auth/login` now set `access_token`/`refresh_token` httpOnly cookies alongside their unchanged JSON response shapes; new `POST /auth/refresh` (cookie-only, sliding renewal) and `POST /auth/logout` (no auth, always 200, clears both cookies) added. `app.ts` now mounts `cors({ origin: process.env.CORS_ORIGIN, credentials: true })` and `cookieParser()` ahead of routes. Backend `PORT` changed from `3000` to `5000` (frees `3000` for the Phase 3 frontend dev server); new env vars `JWT_REFRESH_SECRET` and `CORS_ORIGIN=http://localhost:3000`. Phase 2 backend auth upgrade is feature-complete: `docker compose up -d --build` then register (cookies set) → protected route via cookie jar, no header → refresh (new cookies) → logout (cookies cleared, agent locked out) all work against real Postgres. Also fixed a real bug in Task 1's `JwtTokenService` found by this task's own integration test: `signAccessToken`/`signRefreshToken` signed only `{ userId }` with second-granularity `iat`/`exp`, so two tokens issued for the same user within the same wall-clock second (e.g. register immediately followed by refresh) were byte-identical — silently breaking the sliding-refresh "always a new token" guarantee. Fixed by adding a random `jti` claim (`crypto.randomUUID()`) to every signed token; confirmed the fix by re-running the previously 100%-reproducible failing test 3x green afterward.

Final whole-branch review (Phase 2): approved, ready to merge, with one Important defense-in-depth hardening applied before merge. Access/refresh token isolation previously relied *solely* on `JWT_SECRET`/`JWT_REFRESH_SECRET` being distinct values (nothing in code enforced it) — if a deploy config ever set them equal, a leaked access token could be replayed against `/auth/refresh` indefinitely. Fixed two ways: (1) `JwtTokenService` now signs a `typ: 'access'`/`typ: 'refresh'` claim and `verifyAccessToken`/`verifyRefreshToken` each reject the wrong `typ`, so token-kind separation no longer depends solely on secret distinctness (`tests/unit/JwtTokenService.test.ts`, 2 new "typ backstop" tests using intentionally identical secrets, sabotage-verified); (2) `createApp()` now throws at startup if `JWT_SECRET === JWT_REFRESH_SECRET`, failing loudly at boot instead of silently degrading (`tests/unit/app.test.ts`, new file, 2 tests, sabotage-verified). 100/100 tests passing, typecheck clean.

## Phase 3: Frontend

Stack: Next.js 16 App Router, React 19, Tailwind 4, GSAP 3.15 + @gsap/react, Biome, Vitest.

Layering (dependencies point toward pure logic; the Clean Architecture Mandatory Rule governs the backend, the frontend follows the same spirit): `src/lib/` (framework-light — `types`, `format`, `realm-constants`, `api`, `auth-context`), `src/hooks/` (`use-cultivation-state`, `use-toast`), `src/components/` (presentational + GSAP), `src/app/` (routes + `globals.css`). `api.ts` centralizes `credentials: "include"` and the single-shot refresh retry (`isRefreshing` guard); a 401 that a refresh can't recover throws `"Authentication expired"`, which `use-cultivation-state` maps to a `/login` redirect.

Config: `NEXT_PUBLIC_API_BASE=http://localhost:5000` in `frontend/.env.local` (backend port; the frontend dev server owns `:3000`, which the backend's `CORS_ORIGIN` already allows with credentials). `pnpm-workspace.yaml` allows the `esbuild` build script (pulled in by Vitest) — without it `pnpm <script>`'s deps precheck fails.

Testing: pure-logic unit tests only (`format`, `realm-constants`, and the `api` refresh flow with `fetch` stubbed) — 19 tests, `vitest.config.ts` uses `environment: "node"` + `include: ["src/**/*.test.ts"]`. Animated components are verified by human observation, not snapshots. `pnpm test` exits non-zero on "no test files" (Vitest 3), which only matters before the first test task.

Note on the stats modal: the reference closes it via a click handler on a `<div>` backdrop; that trips this project's Biome a11y rules (recommended `next`/`react` domains are enabled in `biome.json`), so here the backdrop is an accessible sibling `<button>` filling the overlay while the `.popup-panel` (position: relative) stacks above it — same click/keyboard-to-close behavior, lint-clean.

Commands: `cd frontend && pnpm dev` (serves `:3000`), `pnpm test` (Vitest), `pnpm lint` / `pnpm format` (Biome), `pnpm build`. The frontend needs network at build/dev time (`next/font/google` fetches Ma Shan Zheng, ZCOOL XiaoWei, Be Vietnam Pro) and the backend running (`cd backend && docker compose up -d --build`) for any auth/dashboard check.

Verification (Task 14): full gate green (lint, tsc, 19 tests, build). With the backend up on `:5000`, exercised the exact frontend contract via a cookie jar — register sets both httpOnly cookies, `GET /cultivation/state` works cookie-only (no header), breakthrough returns the `{error:{code,message}}` shape when linh khí is insufficient, `/auth/refresh` slides, `/auth/logout` clears, and a post-logout state read is 401. ## Phase 4: Backend Đan Dược

Real backend for the alchemy inventory (the frontend still uses its mock `use-pill-inventory` — wiring is a future task). Spec `docs/superpowers/specs/2026-07-18-dan-duoc-backend-design.md`, plan `docs/superpowers/plans/2026-07-18-dan-duoc-backend.md`.

- **Schema (migration `dan_duoc`):** `Pill` table (definitions live in the DB, seeded via `prisma/seed.ts` / `npm run db:seed`, 8 pills mirroring the frontend catalog — a deliberate departure from the config-in-code `realms.ts` pattern), `InventoryItem` (`@@unique([userId, pillId])`, ownership + quantity), and three new `Character` fields: `cultivationBuffMultiplier`/`cultivationBuffUntil` (active timed buff) and `breakthroughBonusPct` (pending one-shot boost, default 0). `User` gained an `inventory` relation — integration test teardowns now `inventoryItem.deleteMany()` before `user.deleteMany()` (the new FK blocks the old order).
- **Domain:** `domain/pills/pill.ts` (`PillRecord`, `PillEffectKind`, `InventoryEntry`) and `pill.calc.ts` (`applyPillEffect`, pure — one branch per effect kind, buff/boost are refresh/replace not stack/add). `computeLinhKhi` gained an optional `buff` param and integrates **piecewise**: the buffed segment `[lastUpdateAt, min(now, until)]` accrues at `rate × multiplier`, the remainder at `rate` (absent/expired buff ⇒ identical to before). `computeSuccessRate` gained an optional `bonusPct` term added before the max-rate clamp.
- **Ports/infra:** `PillRepository` (`findById`, `listInventory`, atomic `decrementOne` via `updateMany … quantity > 0` + `count === 1`, `seedStarterInventory`) with `STARTER_INVENTORY` constant; `PrismaPillRepository` + `InMemoryPillRepository` fake. `RegisterUserUseCase` takes a `PillRepository` and seeds the starter inventory on registration.
- **Application:** `GetInventoryUseCase` (read-only DTO list); `ConsumePillUseCase` (applicability guards → recompute buffed linh khí → `decrementOne` FIRST so a concurrent double-consume can't apply twice → `applyPillEffect` → optimistic-concurrency persist → returns the cultivation-state DTO). `AttemptBreakthroughUseCase` now uses buffed accrual + adds `breakthroughBonusPct` to the success rate and resets it to 0 on any *resolved* attempt (success or failure; rejection paths leave it). **`GetCultivationStateUseCase` also passes the buff into `computeLinhKhi`** so the frontend's polled state reflects faster accrual (a fix caught by manual verification — the write paths had it but the read path didn't).
- **Presentation:** `GET /pills/inventory`, `POST /pills/consume` (zod `pillId`); error codes `PILL_NOT_FOUND` 404 / `PILL_OUT_OF_STOCK` 409 / `PILL_NOT_APPLICABLE` 400. Wired in `app.ts`.

Manual cookie-jar pass against Docker Postgres confirmed: starter inventory (8 pills, cookie-only), linhKhi consume (+50, qty 5→4), cultivationBuff reflected in `/cultivation/state` (~×1.5 accrual), breakthroughBoost persisted (`breakthroughBonusPct=15`), clearPunishment when not punished → 400, single-unit pill consumed twice → 409.
