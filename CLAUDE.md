# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A cultivation-game (gameplay rebuilt to 100% feature parity with Nhất Niệm Tiêu Dao / 一念逍遥) with a Node/Express backend (`backend/`) and a Next.js frontend (`frontend/`). See `docs/superpowers/specs/` for design docs and `docs/superpowers/plans/` for implementation plans — read the relevant spec before changing game-logic behavior.

## Mandatory Rules

- **Architecture: Clean Architecture.** Backend code is organized into strict layers with dependencies pointing inward only:
  - `domain/` — entities and pure business logic (e.g. linh khí accumulation formula, breakthrough pity/success-rate formula, stage transitions). Zero framework or library dependencies. Defines repository/service interfaces (ports) that outer layers implement.
  - `application/` — use cases that orchestrate domain logic (e.g. RegisterUser, LoginUser, GetCultivationState, AttemptBreakthrough). Depends only on domain interfaces, never on concrete infrastructure.
  - `infrastructure/` — concrete implementations of domain ports: Prisma-backed repositories, JWT token service, bcrypt password hasher, realm config data.
  - `presentation/` — Express routes, controllers, middleware, request validation/DTO mapping. Translates HTTP ↔ use case input/output.
  - A composition root (`src/main.ts` or `app.ts`) wires concrete infrastructure into use cases and controllers.
  - `domain` must never import from `infrastructure` or `presentation`.
- **Comment logic clearly.** Non-trivial business/domain logic (formulas, state transitions, concurrency handling, etc.) must have clear comments explaining the *why* and the mechanics — do not leave complex logic uncommented.
- **Update CLAUDE.md after every task.** When executing an implementation plan, after completing each task, update this file to reflect new architecture, commands, or notes introduced by that task.
- **Use context7 (`ctx7` CLI) before writing library-specific code.** Before using any API from a dependency (Express, Prisma, Next.js, GSAP, jsonwebtoken, zod, etc.), fetch current docs for that library via context7 and cross-check against the exact version pinned in `package.json` — do not rely on training data for library API shape.

Also fixed a real bug in Task 1's `JwtTokenService` found by this task's own integration test: `signAccessToken`/`signRefreshToken` signed only `{ userId }` with second-granularity `iat`/`exp`, so two tokens issued for the same user within the same wall-clock second (e.g. register immediately followed by refresh) were byte-identical — silently breaking the sliding-refresh "always a new token" guarantee. Fixed by adding a random `jti` claim (`crypto.randomUUID()`) to every signed token; confirmed the fix by re-running the previously 100%-reproducible failing test 3x green afterward.

Final whole-branch review (Phase 2): approved, ready to merge, with one Important defense-in-depth hardening applied before merge. Access/refresh token isolation previously relied *solely* on `JWT_SECRET`/`JWT_REFRESH_SECRET` being distinct values (nothing in code enforced it) — if a deploy config ever set them equal, a leaked access token could be replayed against `/auth/refresh` indefinitely. Fixed two ways: (1) `JwtTokenService` now signs a `typ: 'access'`/`typ: 'refresh'` claim and `verifyAccessToken`/`verifyRefreshToken` each reject the wrong `typ`, so token-kind separation no longer depends solely on secret distinctness (`tests/unit/JwtTokenService.test.ts`, 2 new "typ backstop" tests using intentionally identical secrets, sabotage-verified); (2) `createApp()` now throws at startup if `JWT_SECRET === JWT_REFRESH_SECRET`, failing loudly at boot instead of silently degrading (`tests/unit/app.test.ts`, new file, 2 tests, sabotage-verified). 100/100 tests passing, typecheck clean.

## Realm Config in DB + Admin API

Realm/sub-stage tuning data moved out of the hard-coded `REALMS` constant into a DB table so an admin can retune the game at runtime with no redeploy. Spec `docs/superpowers/specs/2026-07-18-realm-config-db-admin-design.md`, plan `docs/superpowers/plans/2026-07-18-realm-config-db-admin.md`. Built on `feat/realm-config-db`.

- **Schema (migration `realm_config`):** new `RealmStage` table (one row per sub-stage, `@@unique([realmMajor, realmSub])`), seeded via `prisma/seed.ts` from the `SEED_REALMS` literal (today's 12×5 balance, values unchanged; 60 rows). `User` gained `role String @default("user")` (`"user"` | `"admin"`).
- **Domain:** `realms.ts` no longer exports a live `REALMS`/`MAX_REALM_*`. It keeps the pure types + `SEED_REALMS` (seed source of truth) and adds `RealmConfigSet` (value object: `getStage`, `realmName`, `maxRealmMajor`, `peakRealmSub(major)`, `clampStage(major, sub)`), `realmConfigSetFromRows`, `flattenRealms`, `defaultRealmConfigSet`. New port `RealmConfigSource { get(): RealmConfigSet }` (synchronous, so use cases keep sync access). `breakthrough.calc.ts` unchanged (`peakRealmSub` was already injected).
- **Ports/infra:** `RealmConfigRepository` (`loadAll` / atomic `replaceAll` via `$transaction([deleteMany, createMany])`) with `PrismaRealmConfigRepository` + `InMemoryRealmConfigRepository` fake. `RealmConfigProvider` (composition-root cache): `ensureLoaded()` (boot-time one-shot, shared in-flight promise), `get()` (sync), `reload()` (after admin writes). `app.ts` mounts an `ensureLoaded`-gate middleware (after `/health`, so health never blocks on the DB). `UserRepository` gained `findById`; `TokenService.signAccessToken(userId, role)` / `verifyAccessToken → { userId, role }`; `JwtTokenService` signs a `role` claim (defaults missing→`"user"` for legacy tokens).
- **Application:** the three cultivation use cases (`GetCultivationState`, `AttemptBreakthrough`, `ConsumePill`) now take a `RealmConfigSource` and read via `config.getStage/maxRealmMajor/peakRealmSub`. `GetCultivationStateUseCase` **lazy-clamps**: an out-of-range `(realmMajor, realmSub)` is clamped to the nearest valid stage and persisted (optimistic-concurrency guard, `lastUpdateAt` untouched) — this fixes the old out-of-range 500. `RefreshAccessTokenUseCase` re-reads the user (`findById`) so a promotion takes effect on the next refresh; `Login`/`Register` sign the token with `user.role`. New `UpdateRealmConfigUseCase` validates invariants (≥1 realm; each realm ≥1 sub-stage; `linhKhiRequired` strictly increasing **within each realm** — NOT across realm boundaries, since the balance resets lower at each new realm, e.g. Phàm Nhân peak 500 → Luyện Khí start 300) then `replaceAll`. Violation → `INVALID_REALM_CONFIG`.
- **Presentation:** `requireAdmin` middleware (after `requireAuth`; non-admin → `FORBIDDEN`). `errorHandler` maps `FORBIDDEN` 403 / `INVALID_REALM_CONFIG` 400. `GET /admin/realms` (current config from the provider cache) + `PUT /admin/realms` (zod-validated full replace, then `provider.reload()` for immediate effect), both behind `requireAuth` + `requireAdmin`.
- **Admin bootstrap:** no self-service endpoint — grant via SQL: `UPDATE "User" SET role='admin' WHERE username='<name>';` then the user re-logs-in so their access token carries `role:admin`.
- **Verification:** `cd backend && npm test` → 171 tests green (unit + integration vs real Postgres). Manual cookie-jar pass against Docker Postgres: non-admin GET/PUT `/admin/realms` → 403; admin GET → config; admin PUT (bump realm 0 sub 0 `linhKhiRequired`) → a fresh player's `/cultivation/state` immediately shows the new value (live reload confirmed); invalid PUT (non-increasing within a realm) → 400 `INVALID_REALM_CONFIG`; a character corrupted to `realmMajor=3, realmSub=9` → `/cultivation/state` returns 200 clamped to `Kết Đan - Viên Mãn` (no 500). Note: the frontend needs no change — it renders stage names from the DTO's `realmName`.

## Admin Pill Catalog

Admin CRUD for the đan dược catalog at runtime — create/edit/soft-disable pills and tune starter grants from `/admin/pills`, replacing seed-file edits. Spec `docs/superpowers/specs/2026-07-20-admin-pill-catalog-design.md`, plan `docs/superpowers/plans/2026-07-20-admin-pill-catalog.md`. Built on `feat/admin-pills` (from `main`).

- **Schema (migration `pill_admin`):** `Pill` gained `active Boolean @default(true)` (soft-disable — never a hard delete: `InventoryItem` rows survive, re-enabling restores holdings) and `starterQuantity Int @default(0)` (new-player grant). The hardcoded `STARTER_INVENTORY` constant is **deleted** — `seedStarterInventory` reads the DB (`active AND starterQuantity > 0`). Seed carries the old quantities (5/3/1/2/1/2/1/2), so registration behavior is unchanged. Re-running `db:seed` overwrites admin edits (upsert full rows) — it's a reset tool, not a routine command.
- **Domain:** `PillRecord` gained the two fields. New pure `pill.validate.ts` (`validatePillDefinition`) — per-effectKind stat requirements (`linhKhi`: amount>0; `cultivationBuff`: multiplier>1 + integer durationSec>0; `breakthroughBoost`: bonusPct>0), non-kind stat fields must be null (no orphans), rarity int 0–4, starterQuantity int ≥0, name/glyph/desc non-empty. Violation → `INVALID_PILL_CONFIG`.
- **Ports/infra:** `PillRepository` gained `listAll()` (includes inactive — admin only), `create`, `update` (full-row by immutable id, `updateMany` count → false on unknown id). `listInventory` filters `pill.active` (relation filter). Prisma impl + fake updated.
- **Application:** `ListPillsAdminUseCase`, `CreatePillUseCase` (validate → `findById` dup check → `PILL_ID_TAKEN`; a concurrent duplicate racing the check hits the PK constraint → 500, accepted for admin-only), `UpdatePillUseCase` (validate → `PILL_NOT_FOUND` on unknown id; enable/disable is just the `active` field). `ConsumePillUseCase` rejects inactive pills with `PILL_NOT_FOUND` *before* `decrementOne` — players can't distinguish disabled from nonexistent, and the unit is never spent.
